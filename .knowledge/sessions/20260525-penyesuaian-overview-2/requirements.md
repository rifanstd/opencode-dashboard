# Requirements: Overview Page Enhancements v2

## Context

Session `20260525-penyesuaian-chart-format-number-cards-overview` implemented 3 charts (TokenUsageChart, CostChart, ModelUsageChart) plus 9 key metric cards on the Overview page. The user now requests 4 refinements: (1) add icons to metric cards, (2) constrain chart grid to 2 columns on desktop, (3) fix chart horizontal alignment within containers, and (4) add a 4th chart showing token composition.

## Functional Requirements

### FR-01: Icons on Key Metric Cards (Right Side)
- **FR-01a:** `SummaryCard` must accept an optional `icon` prop of type `React.ReactNode` (to receive lucide-react icon components).
- **FR-01b:** When `icon` is provided, it must render on the **right side** of the card, with `size={28}` (lucide default prop), `opacity: 0.6`, and `color: var(--text-secondary)` — positioned via flex layout (`justifyContent: space-between` or absolute positioning) so it does not interfere with the card text flow (label top, value middle, subLabel bottom).
- **FR-01c:** The icon must NOT wrap the text content; icon sits in the right margin area, vertically centered relative to the card height.
- **FR-01d:** In `Overview.tsx`, each of the 9 metric cards must receive a lucide-react icon component mapped by its label:
  - Total Tokens → `<Coins />`
  - Input Tokens → `<ArrowDownToLine />`
  - Cache Miss → `<DatabaseZap />`
  - Cache Read → `<HardDrive />`
  - Output Tokens → `<ArrowUpFromLine />`
  - Total Sessions → `<ScrollText />`
  - Providers → `<Server />`
  - Models → `<Cpu />`
  - Estimated Cost → `<DollarSign />`
- **FR-01e:** All 9 icon components must be imported from `lucide-react` at the top of `Overview.tsx`.
- **FR-01f:** If `icon` is not provided (backward compatibility), the card renders as before with no icon — no visual change for other pages using `SummaryCard`.

### FR-02: Chart Grid — Max 2 Columns on Desktop
- **FR-02a:** The charts section grid in `Overview.tsx` must use a fixed 2-column layout on desktop (viewport >= 768px): `grid-template-columns: 1fr 1fr` (each column takes equal width).
- **FR-02b:** On mobile (viewport < 768px), the grid collapses to single column: `grid-template-columns: 1fr`.
- **FR-02c:** All 4 charts (Token Usage, Cost, Model Usage, Token Composition) render in a 2x2 arrangement on desktop: row 1 = TokenUsageChart + CostChart, row 2 = ModelUsageChart + TokenCompositionChart.
- **FR-02d:** The `minmax(360px, 1fr)` pattern must be removed — the grid now has exactly 2 equal-width columns, not auto-fitting.
- **FR-02e:** Gap between grid cells remains `16px` (unchanged).

### FR-03: Fix Chart Horizontal Alignment in Containers
- **FR-03a:** All 4 chart components must have symmetric margins so the chart drawing area (plot area) is horizontally centered within the container card. The visible distance from the left container edge to the Y-axis should roughly equal the distance from the right container edge to the right edge of the plot area.
- **FR-03b:** Use Recharts `<LineChart>` / `<BarChart>` / `<PieChart>` `margin` prop to achieve symmetric centering. Set `margin={{ left: 0, right: 8, top: 5, bottom: 5 }}` on each chart component — reducing the default Recharts left margin (which defaults to ~20–30px) eliminates the rightward shift.
- **FR-03c:** The chart wrapper `<div>` (child of `<ResponsiveContainer>`) must also have no additional left padding that would exacerbate asymmetry. Pure `width: 100%, height: 320` as the inner dimensions.
- **FR-03d:** After the fix, the Y-axis tick labels may clip slightly on the left if the chart container padding is insufficient — the container `padding: 16` (from `chartContainerStyle`) provides enough buffer. No additional width constraint on YAxis is needed.
- **FR-03e:** The margin-based fix applies to TokenUsageChart, CostChart, ModelUsageChart, and the new TokenCompositionChart.

### FR-04: New Chart — Token Composition (Donut/Pie)
- **FR-04a:** Render a **donut chart** using Recharts `<PieChart>` with `innerRadius="60%"` and `outerRadius="80%"`. Apply margin-based centering consistent with FR-03: `margin={{ left: 0, right: 8, top: 5, bottom: 5 }}`.
- **FR-04b:** Show exactly **5 segments** with distinct colors, representing the full token type breakdown:
  1. **Input** — non-cached prompt tokens sent to model (the `input` field)
  2. **Output** — tokens generated by models (the `output` field)
  3. **Reasoning** — thinking/reasoning tokens (the `reasoning` field)
  4. **Cache Read** — tokens served from prompt cache (the `cache` field)
  5. **Cache Miss** — same as Input (non-cached tokens), shown as a separate segment for clarity
- **FR-04c:** Each segment uses a distinct color from the CSS chart palette:
  - Input → `--chart-1`
  - Output → `--chart-2`
  - Reasoning → `--chart-3`
  - Cache Read → `--chart-4`
  - Cache Miss → `--chart-5`
- **FR-04d:** All 5 colors must be visually distinguishable from each other when rendered in the donut.
- **FR-04e:** A **legend** positioned below or beside the donut identifies each segment by name and shows its absolute token count (using `formatNumber()`) and percentage (e.g., "Cache Read — 17.7M (93.1%)").
- **FR-04f:** The center of the donut must display the **total token count** (sum of all 5 segments) formatted with `formatNumber()`, serving as the donut's "total" label.
- **FR-04g:** Tooltip on hover shows segment name, token count (via `formatNumber()`), and percentage of total (1 decimal place).
- **FR-04h:** Data source: `tokenUsage.byDay` aggregated across all days (sum each field: `input`, `output`, `reasoning`, `cache`). Use updated `computeTokenComposition()` from `overviewDataProcessor.ts` (see FR-05) which returns 5 segments.
- **FR-04i:** Handle edge cases: all zeros → show donut with single "No data" segment or a placeholder message; exactly one non-zero segment → show single filled segment with others at zero; no data loaded → "No data available" text.
- **FR-04j:** Chart title: "Token Composition" displayed above the donut (matching the h3 pattern from other charts).
- **FR-04k:** Chart container uses the same `chartContainerStyle` as other charts: `background: var(--bg-secondary)`, `borderRadius: 10`, `padding: 16`, `border: 1px solid var(--border)`.

### FR-05: Update computeTokenComposition to Return 5 Segments
- **FR-05a:** Modify `computeTokenComposition()` in `src/utils/overviewDataProcessor.ts` to return **5 segments**: Input, Output, Reasoning, Cache Read, Cache Miss.
- **FR-05b:** The `DonutSegment` type in `src/types/index.ts` already supports `{ name, value, color }` — no type change needed; the function simply returns 5 items instead of 3.
- **FR-05c:** Segment definitions:
  - Input: sum of `input` field from `byDay`, color `var(--chart-1)`
  - Output: sum of `output` field from `byDay`, color `var(--chart-2)`
  - Reasoning: sum of `reasoning` field from `byDay`, color `var(--chart-3)`
  - Cache Read: sum of `cache` field from `byDay`, color `var(--chart-4)`
  - Cache Miss: same value as Input (non-cached tokens), color `var(--chart-5)`. This means Cache Miss will have the identical `value` as Input — both represent the same underlying data, presented as a separate visual slice.
- **FR-05d:** Handle zero-total edge case: if the sum of all 5 values is zero, return 5 segments all with `value: 0` so the donut renders an empty state gracefully.

## Non-Functional Requirements

- **NFR-01 (Visual Consistency):** The new TokenCompositionChart container must match the existing 3 chart containers in style: same background, border, border-radius, padding (16px), and heading pattern (h3 with fontSize 16).
- **NFR-02 (Responsiveness):** The donut chart must shrink proportionally on mobile. Legend items may wrap below the donut on narrow viewports.
- **NFR-03 (Performance):** The token composition computation must use `useMemo` with `tokenUsage` as the dependency — no recalculation on every render.
- **NFR-04 (Type Safety):** All new code must pass `tsc -b` with the project's strict TypeScript settings (`verbatimModuleSyntax`, `erasableSyntaxOnly`, `noUnusedLocals`, `noUnusedParameters`).
- **NFR-05 (Dependencies):** A new dependency `lucide-react` is introduced for metric card SVG icons. It is lightweight (~5KB gzipped), tree-shakeable, and widely adopted. Recharts remains the sole charting library — no new charting dependencies.
- **NFR-06 (Backward Compatibility):** The `icon` prop on `SummaryCard` is optional — other pages using `SummaryCard` without `icon` must render unchanged.

## Constraints

- **C-01:** Must use existing Recharts library for all 4 charts. No new charting libraries.
- **C-01b:** New dependency: `lucide-react` (MIT license, lightweight SVG icon library) for metric card icons. Must be installed via `npm install lucide-react`.
- **C-02:** Client-side only — no server-side data aggregation. Donut data computed from `tokenUsage.byDay` via `useMemo`.
- **C-03:** Must follow existing project patterns: inline styles (no CSS modules/tailwind), React functional components, `useMemo`/`useState` for derived data.
- **C-04:** TypeScript strict mode: `verbatimModuleSyntax: true`, `erasableSyntaxOnly: true`, `noUnusedLocals: true`, `noUnusedParameters: true`.
- **C-05:** `public/data/` is git-ignored. No JSON files committed.
- **C-06:** The `workers/` directory must remain untouched (legacy).
