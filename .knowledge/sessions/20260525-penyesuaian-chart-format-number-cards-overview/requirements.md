# Requirements: Penyesuaian Chart, Format Number, dan Cards Key Metrics Overview

## Context

The Overview page (`src/pages/Overview.tsx`) currently has 4 sections: Key Metrics (9 cards), Trends (DailyActivityChart + CostTrendChart), Breakdowns (TokenCompositionChart, TopModelsChart, ProjectActivityChart), and Recent Activity (ActivityHeatmap + RecentSessionsList). The user wants to restructure the Overview page: reorder key metrics cards, replace all existing charts with 3 new charts (Token Usage, Model Usage, Cost), improve number formatting (K/M/B suffixes for counts, max 2 decimal places for money), and add `byYear` aggregation to the sync script.

## Functional Requirements

### FR-01: Format Number — Count Values with K/M/B Suffixes
- **FR-01a:** Extend `formatNumber()` in `src/utils/costCalculator.ts` to support billions (B suffix). Values >= 1,000,000,000 (1B) render as `X.XB` (one decimal).
- **FR-01b:** Existing K and M behavior unchanged: < 1,000 renders raw number; 1,000–999,999 renders as `X.XK`; 1,000,000–999,999,999 renders as `X.XM`.
- **FR-01c:** Trailing `.0` must be stripped for all suffixes (e.g., `2M` not `2.0M`).
- **FR-01d:** The function must handle non-finite values by returning `'0'`.
- **FR-01e:** All token counts displayed on cards, chart axes, and tooltips must use `formatNumber()`.

### FR-02: Format Cost — Max 2 Decimal Places
- **FR-02a:** Modify `formatCost()` to always use 2 decimal places regardless of the value magnitude. Remove the existing special case that uses 4 decimal places for values < $1.
- **FR-02b:** Use standard mathematical rounding (half-up) for the second decimal. Example: `$0.2345` → `$0.23`, `$0.235` → `$0.24`, `$0.001` → `$0.00`.
- **FR-02c:** The `$` prefix must be preserved. `$0.00` for zero and non-finite values.

### FR-03: Reorder Key Metrics Cards
- **FR-03a:** Change the card order produced by `computeKeyMetrics()` in `src/utils/overviewDataProcessor.ts` to:
  1. Total Tokens
  2. Input Tokens
  3. Cache Miss
  4. Cache Read
  5. Output Tokens
  6. Total Sessions
  7. Providers
  8. Models
  9. Estimated Costs
- **FR-03b:** Each card must retain its label, value, subLabel, and trend properties.
- **FR-03c:** The Input Tokens card value must equal the sum of `input + cache` tokens across all sessions (Cache Miss + Cache Read combined). SubLabel: `"Prompt + cache tokens"`. This matches the current semantic in `computeKeyMetrics()` line 124: `totalInputTokens + totalCacheTokens`.
- **FR-03d:** The Estimated Costs card value must use `formatCost()` (with the new 2-decimal rule) when cost < $1000, otherwise use `'$' + formatNumber(value)` for larger values.

### FR-04: Remove Old Chart Sections
- **FR-04a:** Remove the entire "Trends" section (lines 218–242 in Overview.tsx) including `DailyActivityChart` and `CostTrendChart`.
- **FR-04b:** Remove the entire "Breakdowns" section (lines 245–268) including `TokenCompositionChart`, `TopModelsChart`, and `ProjectActivityChart`.
- **FR-04c:** Remove the entire "Recent Activity" section (lines 271–281) including `ActivityHeatmap` and `RecentSessionsList`.
- **FR-04d:** Remove unused state variables from Overview.tsx: `activityRange`, `costRange`, `modelSortBy`.
- **FR-04e:** Remove unused `useMemo` computations from Overview.tsx: `tokenComposition`, `dailyActivity`, `dailyCost`, `topModels`, `topProjects`, `recentSessions`, `heatmapData`.
- **FR-04f:** Remove unused imports: `TokenCompositionChart`, `DailyActivityChart`, `CostTrendChart`, `TopModelsChart`, `ProjectActivityChart`, `ActivityHeatmap`, `RecentSessionsList`.
- **FR-04g:** The component files themselves may remain in the codebase (other pages may reference them) — only imports and usage in Overview.tsx are removed.

### FR-05: New Chart — Token Usage
- **FR-05a:** Render a smooth **line chart** using Recharts `<LineChart>` with `type="monotone"` on each `<Line>` (non-sharp curves).
- **FR-05b:** X-axis displays date/timestamp labels. Format depends on filter granularity:
  - Daily: `MMM DD` (e.g., "May 24")
  - Weekly: `MMM DD` (first day of the week)
  - Monthly: `MMM YYYY` (e.g., "May 2026")
  - Year: `YYYY` (e.g., "2026")
  - All: daily format (`MMM DD`) showing all available data points
- **FR-05c:** Y-axis displays token counts formatted via `formatNumber()` (K/M/B suffixes). Must handle empty data gracefully (show zero axis).
- **FR-05d:** Render exactly **5 lines** with distinct colors:
  1. **Input Tokens** (total input = `input + cache` — all tokens sent to the model, cached or not)
  2. **Cache Read** (`cache` field only — tokens served from prompt cache)
  3. **Cache Miss** (`input` field only — non-cached tokens sent to the model)
  4. **Output Tokens** (`output` field only — tokens generated by the model)
  5. **Reasoning** (`reasoning` field only — thinking/reasoning tokens)
- **FR-05e:** Each line must use a different color from the CSS chart palette (`--chart-1` through `--chart-7`). Colors must be visually distinguishable.
- **FR-05f:** A **legend** must be displayed identifying each line by name.
- **FR-05g:** A **filter/selector** controls the time granularity with options: `Daily`, `Weekly`, `Monthly`, `Year`, `All`.
  - `Daily`: uses `tokenUsage.byDay` data directly.
  - `Weekly`: uses `tokenUsage.byWeek` data directly.
  - `Monthly`: uses `tokenUsage.byMonth` data directly.
  - `Year`: uses `tokenUsage.byYear` data from the updated sync script (see FR-10). No client-side year aggregation needed.
  - `All`: shows all data at daily granularity without time filtering — all `byDay` data points.
- **FR-05h:** Tooltip on hover must show the formatted date, line name, and formatted token count (using `formatNumber()`).
- **FR-05i:** Responsive container must maintain aspect ratio. Chart must be usable on mobile (viewport < 768px).

### FR-06: New Chart — Model Usage
- **FR-06a:** Render a **vertical bar chart** using Recharts `<BarChart>`.
- **FR-06b:** X-axis displays model names. Show **at most 4** models. If fewer than 4 models have usage data, show the available count (3, 2, or 1). Must NOT show more than 4.
- **FR-06c:** Models are selected as the top N by **total token count** (sum of `input + output + reasoning + cache` across all sessions globally).
- **FR-06d:** Y-axis displays total token count per model formatted via `formatNumber()` (K/M/B suffixes).
- **FR-06e:** **Simple bar chart** — one bar per model. Each bar has its own distinct color (not stacked, not grouped). Color is per-model, not per-token-type.
- **FR-06f:** A **legend** identifies which color corresponds to which model.
- **FR-06g:** Data source: `tokenUsage.byModel` from `token-usage.json`, which pre-aggregates by model.
- **FR-06h:** Tooltip on hover must show the model name and formatted total token count.
- **FR-06i:** Handle edge cases: empty data (no sessions) shows "No data" message; single model shows 1 bar.

### FR-07: New Chart — Cost
- **FR-07a:** Render a smooth **line chart** using Recharts `<LineChart>` or `<AreaChart>` with `type="monotone"` (non-sharp curves). Single line for cost.
- **FR-07b:** X-axis displays timestamp labels using the same format rules as the Token Usage chart (FR-05b).
- **FR-07c:** Y-axis displays cost values using `'$' + formatNumber(value)` notation (dollar sign prefix + K/M/B suffixes). Examples: `$1.2K` (for $1,200), `$500` (for $500), `$3M` (for $3,000,000), `$0` (for zero). Tooltip shows full `formatCost()` with `$` prefix and 2 decimal places (e.g., `$1234.56`).
- **FR-07d:** A **filter/selector** controls time granularity: `Daily`, `Weekly`, `Monthly`, `Year`, `All` — same semantics as FR-05g.
  - Cost data is **computed from sessions** (not pre-aggregated in token-usage.json).
  - `Daily`: aggregate cost per day from sessions.
  - `Weekly`: aggregate cost per ISO week.
  - `Monthly`: aggregate cost per month.
  - `Year`: aggregate cost per year (client-side, since cost is not in token-usage.json).
  - `All`: all data at daily aggregation without time filtering.
- **FR-07e:** Cost calculation per session uses `calculateCost()` from `costCalculator.ts` with the model's pricing from `buildPricingMap()`. Sessions without a known model_id or pricing contribute $0.
- **FR-07f:** Tooltip on hover must show the formatted date and cost (formatted with `$` prefix and 2 decimal places via `formatCost()`).
- **FR-07g:** Handle empty data: no cost data shows "No cost data" or empty chart with zero axis.

### FR-08: Data Synchronization
- **FR-08a:** The "Total Tokens" key metric card value must equal the sum of all tokens across the Token Usage chart data (when filter is "All" with daily granularity).
- **FR-08b:** The "Estimated Costs" key metric card value must equal the sum of all costs across the Cost chart data (when filter is "All" with daily granularity).
- **FR-08c:** All token-related cards (Total Tokens, Input Tokens, Cache Miss, Cache Read, Output Tokens) must derive from the **same** `tokenUsage.byDay` data source used by the Token Usage chart.
- **FR-08d:** Model count on the "Models" card must match the model data available for the Model Usage chart.

### FR-09: Empty/Error State Handling
- **FR-09a:** If `token-usage.json` has zero entries (no sessions), all charts display appropriate empty states (e.g., "No data available") rather than crashing.
- **FR-09b:** If `sessions.json` is empty, cost chart shows zero cost across all time ranges.
- **FR-09c:** If pricing data is missing for cost calculation, the cost chart must not crash — treat missing pricing as $0.
- **FR-09d:** All charts must handle gracefully when filtered data results in zero data points.

### FR-10: Sync Script — Add byYear Aggregation
- **FR-10a:** Modify `scripts/sync-opencode-data.js` to add a `byYear` array to the token usage export, alongside existing `byDay`, `byWeek`, `byMonth`.
- **FR-10b:** Each `byYear` entry has the shape: `{ date: "YYYY" (string), input: number, output: number, reasoning: number, cache: number }`.
- **FR-10c:** Year key is derived from session `created_at` timestamps: `String(date.getFullYear())`.
- **FR-10d:** `byYear` must be sorted chronologically by date (ascending).
- **FR-10e:** The aggregation loop (lines 340–361) must add a `yearMap` alongside existing `dayMap`, `weekMap`, `monthMap`.
- **FR-10f:** `byYear` must be included in the `tokenUsage` object written to `public/data/token-usage.json`.
- **FR-10g:** TypeScript type definitions must be updated to include `byYear`:
  - `TokenUsageData` interface in `src/utils/dataLoader.ts`: add `byYear` array with the same element shape as `byDay`/`byWeek`/`byMonth`.

## Non-Functional Requirements

- **NFR-01 (Performance):** All data processing (aggregation, filtering) must be client-side only — no additional API calls beyond the existing `fetch()` to static JSON files. Computations must use `useMemo` to avoid recalculation on re-render. The only exception is `byYear` which is pre-computed in the sync script.
- **NFR-02 (Responsiveness):** Charts must be responsive via Recharts `<ResponsiveContainer>`. On viewport < 768px, layout adjusts to single-column. Chart heights should be at least 300px and scale appropriately.
- **NFR-03 (Accessibility):** Each chart container must include an `aria-label` describing its content. Filter controls must use `aria-pressed` for toggle state.
- **NFR-04 (Consistency):** All 3 new charts must share visual styling: same container background (`var(--bg-secondary)`), border radius (10px), padding (16px), border (`1px solid var(--border)`).
- **NFR-05 (Maintainability):** The new time granularity filter concept (Daily/Weekly/Monthly/Year/All) should be encapsulated in a reusable component or at minimum share consistent behavior across both Token Usage and Cost charts.
- **NFR-06 (Type Safety):** All new interfaces, data types, and function signatures must be declared in `src/types/index.ts` and imported with `import type` where required by `verbatimModuleSyntax`.
- **NFR-07 (Bundle Size):** No new npm dependencies. All charts use existing Recharts library. No unused imports.

## Constraints

- **C-01:** Must use existing Recharts library (already a dependency). No new charting libraries.
- **C-02:** Must be client-side only. No server-side data aggregation. All data comes from `public/data/*.json` via `fetch()`.
- **C-03:** Must follow existing project patterns: Zustand for global state (if needed), `useMemo` for derived data, `useState`/`useEffect` for local state.
- **C-04:** TypeScript strict mode: `verbatimModuleSyntax: true`, `erasableSyntaxOnly: true`, `noUnusedLocals: true`, `noUnusedParameters: true`. All new code must pass `tsc -b`.
- **C-05:** ESLint flat config — no lint warnings or errors introduced.
- **C-06:** `public/data/` is git-ignored. No JSON files committed.
- **C-07:** The `workers/` directory is empty and must remain untouched (legacy).
- **C-08:** Sync script modifications must be backward-compatible: running `npm run sync` must still produce all existing JSON files unchanged in structure, only adding the new `byYear` field to `token-usage.json`.
