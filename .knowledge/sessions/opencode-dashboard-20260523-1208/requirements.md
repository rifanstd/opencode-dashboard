# Requirements: Opencode Dashboard

## Context

Build a client-side dashboard website to visualize opencode (AI coding assistant) usage analytics from local data. The dashboard runs in the browser and reads exported JSON data files that are generated by a companion Node.js script. This hybrid approach allows the dashboard to work in any browser while still accessing local opencode data.

**Architecture:**
1. A Node.js script (`scripts/sync-opencode-data.js`) runs on the user's machine
2. The script auto-detects opencode data locations and reads the SQLite database, auth.json, models.json, agents, skills, and logs
3. The script exports all data to JSON files in `public/data/`
4. The dashboard reads these JSON files via standard HTTP requests
5. A "Sync" button in the dashboard triggers the script to re-export fresh data

**Data Sources (auto-detected by the script):**
- `~/.local/share/opencode/` — SQLite database, auth.json, logs
- `~/.cache/opencode/` — models.json
- `~/.config/opencode/` — agent definitions, skills lock file

The primary data source is an SQLite database (`opencode.db`, ~196MB) containing session metadata, token usage, costs, messages, parts, projects, and other relational data.

## Functional Requirements

- **FR-01 Overview Dashboard**: Display summary cards showing at minimum: total number of sessions, total tokens broken down by type (input / output / reasoning / cache), total estimated cost, most frequently used model, and count of active projects. Each metric must be derived from the loaded local data.
- **FR-02 Token Usage Analytics**: Provide interactive breakdowns of token consumption per model, per provider, per project, and per time granularity (day / week / month). Breakdowns must be presented using data visualizations (charts). Users must be able to switch between time granularities.
- **FR-03 Sessions List**: Render a table or grid of all sessions from the `session` table. Columns must include: title, project, model, tokens (total), cost, and date. The list must support text search (across title, project, model), filtering (by project, model, date range), and sorting (by date, cost, tokens).
- **FR-04 Session Detail**: Provide a drill-down view for a selected session. The view must display the session's messages, message parts, tool calls (if any), and a detailed token breakdown (input / output / reasoning / cache). Navigation from the Sessions List to Session Detail must be possible.
- **FR-05 Providers**: Read `auth.json` and display a list of connected API providers. API key values must be masked (e.g., shown as `sk-...xxxx`). Each provider entry must show its connection/status state if determinable from the file.
- **FR-06 Models**: Read `models.json` and display a list of available models. Each model entry must show: model ID, capabilities, pricing information, and context window limits.
- **FR-07 Agents**: Read all `*.md` files from the agents directory and display a list of defined agents. Each agent entry must show the agent name and a description/summary extracted from the markdown content.
- **FR-08 Skills**: Read `skills-lock.json` and display a list of installed skills. Each skill entry must show: skill name, installed version, and source/origin.
- **FR-09 Logs/Activity**: Read log files from the `log/` directory and display their contents. The view must support filtering by log level (e.g., INFO, ERROR, WARN) and by date range derived from log file names or timestamps within the logs.
- **FR-10 Cost Estimation**: Calculate and display estimated costs for sessions based on token usage and per-model pricing from `models.json`. The cost displayed in the Overview Dashboard, Sessions List, and Session Detail must use this calculation.
- **FR-11 Auto-Detect Data**: A Node.js script must auto-detect opencode data folders on the user's machine by checking common paths (`~/.local/share/opencode/`, `~/.cache/opencode/`, `~/.config/opencode/` on Unix; `%USERPROFILE%\.local\share\opencode\`, etc. on Windows). If auto-detection fails, the script must accept manual path arguments.
- **FR-12 Data Export Script**: The Node.js script must read the SQLite database and all other data files, then export them as JSON files to `public/data/`. The export must include: sessions, messages, parts, projects, token usage aggregations, providers, models, agents, skills, and logs. The script must handle the ~196MB database efficiently.
- **FR-13 Sync Button**: The dashboard must display a "Sync" button in the top bar. Clicking it must trigger the Node.js script to re-export data and refresh the dashboard with the latest data. The sync process must show a loading state.
- **FR-14 Browser Support**: The dashboard must work in all modern browsers (Chrome, Firefox, Safari, Edge). No browser-specific APIs (like File System Access API) should be required.
- **FR-15 Dark Theme Design**: The dashboard must use a dark color theme inspired by the official opencode website (opencode.ai). The theme should be consistently dark but not overly gloomy — using deep navy/blacks as the base with subtle blue/purple accents, clean typography, and sufficient contrast for readability. All components (sidebar, cards, tables, charts, forms) must adhere to this theme. No light mode toggle is required.

## Non-Functional Requirements

- **NFR-01 Privacy**: All data must remain client-side. No data may be transmitted to any external server or API.
- **NFR-02 Performance**: The dashboard must remain responsive while loading and querying the ~196MB SQLite database. Database queries should complete within 2 seconds for typical aggregations on a modern desktop machine.
- **NFR-03 Browser Support**: The dashboard must work in all modern browsers (Chrome, Firefox, Safari, Edge). No browser-specific APIs should be required.
- **NFR-04 Responsiveness**: The UI must be usable on viewport widths from 1280px to 1920px (desktop-focused, given the data source is local files on a developer workstation).
- **NFR-05 Error Resilience**: The application must gracefully handle missing or corrupted data files, displaying user-friendly error messages without crashing.
- **NFR-06 Maintainability**: Code must be written in TypeScript with strict type checking enabled. All new dependencies must be justified in the implementation plan.
- **NFR-07 Visual Design Consistency**: The dark theme must be applied consistently across all pages and components using CSS custom properties (variables). Colors, spacing, typography, and component styles must follow a unified design system documented in the CSS variables. Chart colors must be chosen to remain visible and aesthetically pleasing on dark backgrounds.

## Constraints

- **Tech Stack**: React 19, TypeScript, Vite (existing project). No backend server.
- **Module Syntax**: `verbatimModuleSyntax: true` — only emit `import`/`export`. Use `import type` for type-only imports.
- **Erasable Syntax**: `erasableSyntaxOnly: true` — avoid `enum`, `namespace`, parameter properties, etc.
- **Build Strictness**: `noUnusedLocals`, `noUnusedParameters`, `noFallthroughCasesInSwitch` are all `true`. Unused variables and switch fallthroughs will fail the build.
- **Data Access**: Node.js script reads local files and exports to JSON. Dashboard reads JSON via HTTP.
- **SQLite**: Node.js script uses native `sqlite3` or `better-sqlite3` to read `opencode.db`. Dashboard does not need SQLite in browser.
- **Data Scattered Across 3 Directories**: The Node.js script auto-detects all three directories and exports their data.
- **No Test Runner**: The project has no test runner configured; testing strategy is out of scope for this requirements document.
- **ESM Only**: `package.json` sets `"type": "module"`.

## Open Questions

- **? OQ-01 Layout Architecture**: Should the dashboard use a single-page layout with a sidebar navigation, or a multi-page layout with route-based navigation? This affects routing library choice and component structure.
- **? OQ-02 Auto-Detect Strategy**: How should the Node.js script auto-detect opencode data folders? Should it check environment variables, common paths, or both? What should happen if auto-detection fails?
- **? OQ-03 Sync Mechanism**: How should the "Sync" button trigger the Node.js script? Should it use a Vite plugin, a separate HTTP endpoint, or execute the script directly via a child process?
- **? OQ-04 Charting Library**: Which charting library should be used for token usage analytics? Options include Recharts, Chart.js, D3, or others. Selection criteria include bundle size, TypeScript support, and React integration.
- **? OQ-05 State Management**: What state management approach should be used for holding the loaded data and UI state? Options include React Context, Zustand, Jotai, or Redux. Given the data size, immutability and performance are concerns.
- **? OQ-06 Data Refresh**: How should the dashboard handle data updates while the app is open? Should the "Sync" button re-run the export script and reload the page, or should it fetch new JSON files dynamically?
- **? OQ-07 Cost Calculation Precision**: What rounding rules and currency display format should be used for cost estimation? Should costs be shown in USD only?
- **? OQ-08 Agent Markdown Parsing**: How should agent descriptions be extracted from `*.md` files? Should the app parse frontmatter, use the first paragraph, or display the full markdown content?
