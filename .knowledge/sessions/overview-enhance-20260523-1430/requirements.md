# Requirements: Overview Page Enhancement

## Context

The opencode-dashboard Overview page (`src/pages/Overview.tsx`) currently displays 8 basic summary cards (Total Sessions, Input Tokens, Output Tokens, Reasoning Tokens, Cache Tokens, Estimated Cost, Most Used Model, Active Projects) in a simple grid layout. The user wants the Overview page to be more substantial, high-quality, and insightful — transforming it from a static summary into a rich dashboard that tells a story about opencode usage at a glance.

The dashboard is client-side only, loading static JSON files generated by a sync script. All data computation must happen in the browser. The project already has a dedicated Token Usage page with charts; the Overview should provide a high-level synthesis without duplicating the detailed drill-down experience.

## Functional Requirements

- **FR-01 — Enhanced Key Metrics:** The Overview page shall display a top "Key Metrics" section with summary cards for: Total Sessions, Total Tokens (sum of input + output + reasoning + cache), Estimated Total Cost, Most Used Model, Active Projects, Total Messages, and Sessions in Last 7 Days. Each card shall show the primary value and, where applicable, a contextual sub-label (e.g., "+12 vs last 7 days" if computable).

- **FR-02 — Token Composition Chart:** The page shall include a donut chart showing the proportional breakdown of input, output, reasoning, and cache tokens as percentages of the total token count.

- **FR-03 — Daily Activity Trend:** The page shall include an area chart showing total token usage per day. The chart shall use stacked areas for input, output, reasoning, and cache tokens. A time range selector shall allow the user to choose between 7 days, 30 days, 90 days, or all time. The default shall be 30 days.

- **FR-04 — Cost Trend Chart:** The page shall include a line or area chart showing estimated daily cost. Cost shall be computed client-side from session token data and model pricing. A time range selector shall allow the user to choose between 7 days, 30 days, 90 days, or all time. The default shall be 30 days.

- **FR-05 — Top Models Breakdown:** The page shall include a horizontal bar chart displaying the top 5 most-used models. A toggle shall allow the user to rank by either total token count (input + output + reasoning + cache) or estimated total cost. Each bar shall be segmented by token type. The default ranking shall be by token count.

- **FR-06 — Project Activity Breakdown:** The page shall include a vertical bar chart displaying the top 5 most active projects ranked by total token count. Projects with no associated sessions shall be excluded.

- **FR-07 — Recent Sessions List:** The page shall include a "Recent Sessions" section listing the 10 most recently created sessions. Each row shall display: session title, model name, total tokens, estimated cost, and creation date. Each row shall be clickable and navigate to `/sessions/:id`.

- **FR-08 — Activity Summary Stats:** The page shall compute and display summary statistics derived from session `created_at` timestamps: sessions created in the last 7 days, sessions created in the last 30 days, and average tokens per session (total tokens ÷ total sessions).

- **FR-14 — Tool Call Frequency Metric:** The page shall display a summary metric showing the total number of tool calls derived from the `parts` data (where `type === 'tool'`). This shall appear as a summary card in the Key Metrics section with a sub-label showing tool calls in the last 7 days if computable.

- **FR-15 — Activity Heatmap:** The page shall include an activity heatmap showing message distribution by day-of-week (Mon–Sun) and hour-of-day (00–23). The heatmap shall use color intensity to represent message volume. Data shall be computed from the `messages` array using `created_at` timestamps.

- **FR-16 — Period-over-Period Comparisons:** Summary cards shall display period-over-period comparisons where applicable. For example: "+5 vs last week" for sessions, "+12% vs last week" for tokens. Comparison windows shall be: 7 days vs previous 7 days, 30 days vs previous 30 days. The comparison shall be shown as a sub-label with an up/down indicator and percentage or absolute difference.

- **FR-09 — Client-Side Data Computation:** All visualizations and metrics shall be computed entirely in the browser using existing `dataLoader.ts` functions (`loadOverviewStats`, `loadTokenUsage`, `loadSessions`, `loadMessages`, `loadModels`) and existing `costCalculator.ts` utilities. No modifications to the sync script or new JSON endpoints are permitted.

- **FR-10 — Sectioned Dashboard Layout:** The page shall be organized into clearly separated sections using a responsive grid layout:
  1. Key Metrics (top row, 4–8 cards)
  2. Trends (middle, full-width or 2-column grid: Daily Activity + Cost Trend)
  3. Breakdowns (middle, 2-column grid: Token Composition + Top Models / Project Activity)
  4. Recent Activity (bottom, full-width list)

- **FR-11 — Consistent Chart Styling:** All charts shall use the existing CSS variable color palette (`--chart-1` through `--chart-4`) and tooltip styling consistent with the existing `TokenChart` component. Chart heights shall be uniform (minimum 280px, maximum 360px) within each section.

- **FR-12 — Loading and Error States:** The page shall display a loading indicator while data is being fetched and computed. If any required data source fails to load, an `ErrorMessage` component shall be displayed, and partial data shall not be rendered in a broken state.

- **FR-13 — Responsive Behavior:** On viewports narrower than 768px, all multi-column sections shall collapse to a single column. Chart containers shall remain readable without horizontal scrolling.

## Non-Functional Requirements

- **NFR-01 — Performance:** The Overview page shall complete all data fetching, computation, and initial render of all sections within 2 seconds on a dataset of up to 10,000 sessions and 100,000 messages on a modern desktop browser.

- **NFR-02 — Responsiveness:** The layout shall adapt gracefully from 320px (mobile) to 1920px (desktop) viewport widths. No section shall require horizontal scrolling to view its content.

- **NFR-03 — Interactivity:** All charts shall support hover tooltips showing precise values. The Recent Sessions list rows shall have visible hover states and cursor pointers to indicate clickability.

- **NFR-04 — Accessibility:** All charts shall include descriptive `title` or `aria-label` attributes. The page heading structure shall remain logical (`h1` for page title, `h2` for section titles).

- **NFR-05 — Bundle Size:** The enhanced Overview page shall not introduce new runtime dependencies. Any new components shall be tree-shakeable and shall not increase the production bundle size by more than 15KB gzipped.

- **NFR-06 — Type Safety:** All new components and data transformations shall be fully typed with TypeScript. The build must pass `tsc -b` with zero type errors under the existing strict configuration.

## Constraints

- **Tech Stack:** React 19, TypeScript 5.x, Vite, Recharts, Zustand, React Router v7. No additional charting or UI libraries may be added.

- **TypeScript Strictness:** Must comply with `verbatimModuleSyntax: true`, `erasableSyntaxOnly: true`, `noUnusedLocals: true`, `noUnusedParameters: true`, `noFallthroughCasesInSwitch: true`. Use `import type` for type-only imports.

- **Data Architecture:** Client-side only. All data comes from static JSON in `public/data/`. Must use existing `dataLoader.ts` functions. Must not modify `scripts/sync-opencode-data.js` or add new backend endpoints.

- **Component Reuse:** Must reuse existing `SummaryCard`, `ErrorMessage`, and `TokenChart` components where possible. New chart wrappers are permitted if they extend `TokenChart` or follow its patterns.

- **Styling:** Must use existing CSS variables (`--bg-primary`, `--bg-secondary`, `--text-primary`, `--text-secondary`, `--text-muted`, `--border`, `--chart-1`…`--chart-4`). No inline CSS-in-JS libraries; use inline `style` props or CSS modules already in use.

- **No Test Runner:** The project has no test framework. Acceptance criteria must be verifiable through manual inspection and build success.

## Additional Ideas (User-Approved)

The user is open to additional ideas and new data files if they enhance the dashboard. Consider:
- **Model Provider Distribution:** A pie/donut chart showing token usage or cost distribution across providers (OpenAI, Anthropic, Google, etc.).
- **Session Length Distribution:** A histogram showing the distribution of session lengths (number of messages per session).
- **Cost Efficiency Metric:** Average cost per 1K tokens, or cost per session, to track spending efficiency over time.
- **Most Active Agents:** If agents are linked to sessions, show which agents are most frequently used.
